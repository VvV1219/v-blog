## 关 1 · 浏览器池

### 🎬 事故题 1：为什么不能「每请求 launch 一个浏览器」？

1. **慢 = 冷启动**：`launch()` 是启动一个真 Chrome 进程（加载内核、初始化渲染引擎、建通信通道），要 **1~2s**。每次都白交这份"开机税"。池化 = 开一次反复用，把税摊掉。
2. **资源爆炸**：一个 Chrome 实例 **几百 MB**。100 并发 = 同时几十 GB → 服务器 OOM 被杀。
3. **僵尸进程**：截图中途报错，`close()` 没执行到，Chrome 关不掉，赖在内存里。请求越多僵尸越多 ← 这才是"泄漏"的真身。

→ **池子干三件事：控制总数（最多 N 个）+ 复用（开一次用很多次）+ 兜底回收（报错也保证还回/重建）。**

### 🎬 事故题 2：池满了，第 6 个请求怎么办？

用**等待队列（waiter queue）**：第 6 个挂起等着，谁先还回来就给谁。但排队必须配两道闸，否则被压测打穿：

| 闸 | 作用 | 值 |
|---|---|---|
| **队列长度上限** | 排队 > 20，第 21 个直接拒绝（服务繁忙），别堆爆内存 | `maxWaitingQueueSize` 20 |
| **单个等待超时** | 排队 > 30s 还没轮到，主动放弃报超时，踢出队列 | `acquireTimeout` 30s |

→ **完整答案 = 等待队列 + 长度上限 + 等待超时**，缺一个都不及格。

### 🎬 事故题 3：实例都健康没报错，但内存只涨不降？

Chrome **天生"越用越胖"**（累积缓存 / DOM 残留 / JS 堆），即使没报错也修不好，只能**定期杀了重开**。所以要给每个实例记「服役次数」，超过上限（参照值 50）就强制销毁重建——**实例轮换**。

两种"回收"必须分清（重点）：

| | 触发条件 | 治什么病 | 代码位置 |
|---|---|---|---|
| **健康检查回收** | 探测发现**坏了/卡死** | 僵尸、崩溃、无响应 | 定时器 `_performHealthCheck` |
| **实例轮换（按次数）** | 服役次数**到上限**（哪怕没坏） | 健康膨胀、慢性泄漏 | `release()` 里 `usageCount >= max` |

→ 一句话：**坏的靠"探"出来杀；没坏但老了的，靠"数"出来换。** 两条独立的线。

### 📖 关 1 读码骨架（BrowserPool.js）

**三个核心数组（命根子，要记）：**

- `instances` = 全员花名册（借出的 + 空闲的）
- `available` = 替补席（当前空闲可借）
- `waiting` = 排队等实例的请求
- **借 = 从 available 移除；还 = 放回 available。** 池子就是在搬这三个数组。

**`acquire()` 借实例 = 严格四级降级（核心，要记）：**

1. 有空闲 → 直接借（`available.shift`）— 最划算，复用
2. 没满 → 新建一个借出（`createInstance`）— 扩容
3. 队列也满 → 直接拒绝（`throw QueueFullError`）— 长度上限闸
4. 否则 → 进队列排队（`Promise + setTimeout`）— 等待超时闸；超时先把自己从 `waiting` splice 删掉再 reject

**`release()` 还实例 = 放回前过两道安检：**

1. 服役到上限 → 销毁 + 补新，`return`（不放回）← 实例轮换
2. 连接已断 → 销毁 + 补新，`return`
3. 都通过 → `available.push` + `processQueue()`

**`processQueue()` = 借还的粘合剂：** 只要"有空闲"且"有人排队"就配对发放。任何"多出一个空闲实例"的时刻（release 后、新建后、健康检查补完后）都要调它，否则排队的人被饿死。

## 关 2 · 请求的一生

### 分层边界（核心，最易混）

| 层 | 职责一句话 | 管什么 |
|---|---|---|
| **routes** | 收 req、调 service、统一包装响应/错误 | 只碰 req/res，不含业务 |
| **services** | 【决策层】要不要真去截 | 缓存 / 去重 / 熔断 / 限流 |
| **executor** | 【执行层】编排截图步骤 | 借实例→借页面→拦截→goto→截图→还 |
| **core（池子）** | 【资源层】浏览器够不够、坏没坏 | 借还 / 排队 / 超时 / 回收 / 轮换 |

→ **记牢边界：services 管"要不要干活"，core 管"干活的工具够不够、坏没坏"。** 别把"清理僵尸/排队/超时"算进 services——那是 core 的事。

### 🎬 事故题：一个请求从进到出，经过谁？

把 service 拆成两段：

**第一段 `takeScreenshot` = 门卫层（快速拒绝 + 硬兜底）：**

1. 熔断器 `canExecute()` 断了 → 拒 ✋
2. 限流 `concurrent >= 15` 满 → 拒 ✋
3. **预估超时提前拒绝**：按队列长度估等待时间，铁定超时就拒 ✋
4. `Promise.race([真活, 45s硬超时])` 包一层 ← **Node 超时经典手法，务必记**

**第二段 `_executeScreenshot` = 干活层：**

1. `cache.get` 命中 → 返回缓存 ✋
2. `deduplicator.hasPending` 有人在截 → 搭车 `waitForExisting` ✋
3. `createPending` 登记"我开始截了"
4. `browserPool.acquire` ← 到这才向 core 借实例
5. `acquirePage` → 拦截资源 → goto → 截图
6. `cache.set` 存缓存 → `deduplicator.resolve` 通知搭车的人
7. `circuitBreaker.recordSuccess/Failure` 把成败记给熔断器

→ **三个 ✋ 关键认知：大部分请求在 service 层就被缓存/去重/熔断挡掉，根本不开浏览器。** 点出这句 = 抓住了性能要害。

### 去重的本质

并发请求同一 URL：只有第一个（A）真去 `acquire` 截图；后来的（B）在 `hasPending` 处停下、转去等 A 的 Promise，**拿到 A 截的同一张图**。去重 = 把 N 个并发相同请求**压成 1 次真实工作**（不是"拦住"，是"共享同一个 Promise 结果"）。
为什么 B 不会也登记成 A？**Node 单线程**，A 的"检查→登记"不被打断。

### 架构现实主义（重要）

很多实现其实没真用独立的 `ScreenshotExecutor`，把"借实例→截图"直接写进 `_executeScreenshot`（所以该方法很长）。把 executor 单独拆出来，单一职责上更优雅。→ 生产代码分层不一定干净，能看出它哪可以更好才是真懂。

## 关 3 · 三层缓存 + 去重

### 三层缓存，各堵一种浪费（核心表，要背）

| 层 | 堵的是哪种浪费 | 命中条件 | 比的粒度 |
|---|---|---|---|
| **去重** | N 个**并发**的相同 URL 重复截图 | 同一瞬间、同 URL | 同一瞬间 |
| **截图缓存** | **跨时间**重复截整个相同页面 | 之后、同 URL | 整页 URL |
| **资源缓存** | 不同页面**共用的静态资源**（js/css/字体）被重复下载 | 不同 URL、相同子资源 | 页面内部的零件 |

→ 一句话：**去重比"同一瞬间"；截图缓存比"整页 URL"；资源缓存比"页面里的零件"。** 粒度从粗到细。
→ **资源缓存是常被忽略的第二层**：同站截 10 个页面，公共 `react.bundle.js` 不必下载 10 遍，第一次下完存内存，后面 9 次直接喂。

### LRU 的 O(1)（重点）

普通哈希表能 O(1) 查，但 LRU 难在「O(1) 维护谁最近用过」。**JS `Map` 记插入顺序**（有序哈希表），靠两招做到双 O(1)：

1. **`get` 命中 → `delete` + `set` 挪到末尾** → 末尾永远最新，开头永远最旧（用过就续命）
2. **淘汰 → `map.keys().next().value` 取开头（最旧）删掉** → 不用遍历

→ 非 JS 语言答：**哈希表 + 双向链表**（哈希表 O(1) 查，链表 O(1) 挪/删）。JS 的 Map 内置就是这个有序哈希表。

**演练**：`maxEntries=3`，`set A,B,C` → `[A,B,C]`；`get(A)` → `[B,C,A]`（A 挪末尾续命）；`set(D)` 踢开头 → 踢 **B**，得 `[C,A,D]`。**没有那次 get(A) 的话，被踢的就是 A。**

### 容量 vs 新鲜度（两套独立淘汰）

- **maxEntries**：最多存几条，满了踢**最久没用**的（LRU，看**位置**）
- **ttl**：保质期（固定时长）；**age = now − timestamp**（存了多久）；`age > ttl` → 过期，踢（看**时间**）
- 一条可能位置很新（刚 get 过），但 age 超 ttl 照样判过期 → 时间和位置是两把独立的尺子

### 设计洞察：LRUCache 是基类（模板方法）

截图缓存 + 资源缓存**共用同一个 `LRUCache` 引擎**，各自继承重写两个钩子：

- `generateKey()` — key 怎么算（截图用 url+尺寸+格式；资源用资源 URL）
- `getEntrySize()` — 大小怎么算（截图看条数 maxEntries；资源看字节数 maxSize 100MB）

→ **通用机制（LRU 淘汰）抽基类，各缓存不同的部分（key/size）留子类重写。** 加资源缓存的最优做法：把现有 LRU 抽成基类复用，别复制。

## 关 4 · 容错三件套

### 1. 熔断器（CircuitBreaker）

**为什么需要**：目标网站挂了（每次卡 45s 才失败）→ 坏请求把 3 个实例全占满 → 正常请求借不到实例，**一个外部站拖垮整个服务（雪崩）**。
→ 反直觉关键认知：**「慢的失败」比「快的失败」可怕**，因为它死占资源。容错最怕的不是错，是慢。
**熔断在拦什么**：既然这个站最近一直失败，就别再派实例去送死 → **直接快速拒绝，保住资源**。用"快速失败"换"保住服务"。

**三态（必背）：关闭=通 / 打开=断 / 半开=试**（注意 OPEN 是坏状态！）

```text
关闭CLOSED ──连续失败达阈值(5)──► 打开OPEN ──冷静期(30s)到──► 半开HALF_OPEN
   ▲                                                              │试探成功→关闭
   └──────────────────────────────────────────────────────────────┘试探失败→退回打开
```

**三个易被追问的细节：**

1. **失败是「连续」不是「累计」**：`recordSuccess` 第一行 `failures=0`，成功一次就清零。阈值 5 = 连续 5 次。
2. **状态转换是「懒」的**：OPEN→HALF_OPEN 没有定时器，是下个请求调 `canExecute()` 时顺手检查冷静期到没。整个熔断器无 `setInterval`，请求驱动。
3. **熔断瞬间触发自动恢复**：`recordFailure` 达阈值断开后调 `onCircuitOpen()` 回调 → 接到 service 的 `_triggerAutoRecovery`。熔断器只管"断"，喊一嗓子让 service 去"修"。

### 2. 自动恢复（_triggerAutoRecovery）

健康检查（关1，渐进：坏一个换一个，后台常跑） vs 自动恢复（关4，核弹：熔断那刻整池推倒重建）——两种粒度。
做法：`pool.destroy()` → 等 2s 释放 → `new BrowserPool()` 重建 → `circuitBreaker.reset()` 掰回关闭。
防呆：开头 `if (_isRecovering) return`，防雪崩时重复重建（重操作要加"我在做了"锁）。

### ⚠️ 这个设计的真实软肋

熔断器**分不清失败原因**：目标网站挂了 vs 自己池子烂了，在它眼里一样，都 `failures++` → 一律重建池子。**于是会出现"目标站挂了却把自己池子拆了重建"的浪费。**

- 它敢这么写的算盘：重建是便宜赌注（赌错只亏 2s），真正挡外部雪崩的是熔断的"快速拒绝"，重建只是额外赌内部问题。
- **更好的设计 = 错误分类**：外部错误（目标站超时/404/DNS）不该重建、甚至不该计入熔断；只有内部错误（浏览器启动失败/protocol error/实例崩）才触发重建。
- → 可在 executor 失败时先判 error 类型，分开处理。

### 3. 分层超时

**单层总超时（如只设 45s）的毛病**：它分不清卡在哪一步。请求卡在"借实例"借不到时（池子堵死，本该当场放弃），却会傻等满 45s 才报错 → 用户白等 45s、系统资源被占 45s。**把"该 1 秒放弃的事"硬拖成 45s。**

**分层 = 给每步配符合常理的耐心**，哪步异常就在哪步快速放弃：

```text
借实例 acquireTimeout 10s   ← 借不到=池子堵死，10s 就放弃
等网络空闲 networkIdle 15s  ← 页面加载本该十几秒
请求硬超时 requestHard 45s  ← 最外层兜底，前面全失灵也一定掐断
```

**两原则：**

1. 分层能在"最早可判定无望"的地方止损；单层只能在终点止损。
2. 层级必须**由内向外递增**（内层操作超时 < 外层总超时），否则内层超时形同虚设。关 2 的"预估必超时提前拒绝"是此思想极致。

→ **一句话总结**："超时是分层的——该快速失败的那步快速失败，该兜底的最外层兜底，不让注定失败的请求白占资源到最后一刻。"

## 关 5 · 集群多进程

### I/O vs CPU（前置概念）

- **I/O = 等外部**：等网站加载、等 Chrome 渲染、读写磁盘。CPU 在等，慢。**单线程靠事件循环能高效并发地"等"成百上千个**（非阻塞）。
- **CPU = 自己算**：base64 编码、图片 Buffer、JSON 序列化。**单线程一次只能算一个，会堵**。

### 为什么要集群

Node 单线程 = 同一时刻只用 1 个核。8 核机器单进程跑 → 7 个核闲着，浪费。
**痛点不是 I/O（单线程擅长），而是截图里夹着 CPU 密集的同步活（base64 编码大图）——它独占单线程那几十毫秒，堵死所有并发请求。**
→ `cluster` fork 多 worker：各是独立进程/事件循环/核，一个 worker 算 base64 卡住时别的照常服务。CPU 活摊到多核，吞吐翻倍。
→ 一句话答："Node 单线程擅长 I/O，但截图有 base64 这类 CPU 活会阻塞事件循环；cluster 把它分摊到多核，避免一个重活堵死全部。"

`server.js`：master `fork(WORKERS || numCPUs)` 个 worker，共享同一端口；每个 worker 各跑一遍 `startServer`（各自 `new BrowserPool/Cache`）；worker 崩了 master 1s 后补一个。

### 核心代价：多进程后状态「碎片化」

进程间**内存不共享**。前面学的池子/缓存/去重全是**进程内**内存状态 → 多进程后各 worker 各一份，互不知情：

- **缓存命中率下降**：同一 URL 最坏要被 8 个 worker 各截一次才"全员缓存"。本该 1 次干了 8 次。
- **去重跨不了进程**：A、B 同时收到同一 URL，各自都以为没人在截，各截一次。去重只在单进程内有效。

### 解法：无状态服务 + 共享存储

把**缓存、去重记录**从进程内存挪到进程外的 **Redis**（所有 worker 都连）→ 共享一份，命中率回来，去重跨进程生效。
→ 大思想 **「无状态服务」**：worker 自己不存关键状态，状态放外部共享存储，这样加进程/加机器都行为一致 = **水平扩展**的基石。
→ 但**浏览器池搬不进 Redis**（Chrome 实例绑进程），只能每 worker 一份。**能搬的是"数据状态"，搬不了的是"进程绑定的资源"。**

### 实测：进程数 N → 同一 URL 最坏被截 N 次

cluster 分叉做法：主进程 fork N 个 worker（`WORKERS` 环境变量控制），worker 跑 `startWorker()`；`cluster.on("exit")` + `shuttingDown` 守卫做崩溃自愈。

**实测同一 URL 连发 4 次的耗时（慢=真截图，快=缓存命中）：**

| | 真截图次数 | 现象 |
|---|---|---|
| WORKERS=1 | **1 次** | 第1次慢(4s)，后3次全命中(~3ms)—— 一份缓存 |
| WORKERS=2 | **2 次** | 第1、2次都慢(各worker各截一遍)，第3、4次才命中 —— 缓存被切成2份 |

→ **进程数 N → 同一 URL 最坏被截 N 次**，缓存命中率随进程数下降。这就是多进程必须配 Redis 共享缓存/去重的原因。
→ 教训：选验证信号要选对。"开始截图"日志打在 cache.get **之前**，命中也会打，**不能**用它数缓存miss；**用响应耗时**才区分得开。

## 附录 · 基础概念（关 5 衍生，高频）

### 进程 vs 线程

| | 进程 process | 线程 thread |
|---|---|---|
| 是什么 | OS 分配**资源**的单位 | 进程内的**执行**单位（真正干活） |
| 内存 | 各进程**独立，不共享** | 同进程多线程**共享**内存 |
| 通信 | 麻烦（IPC/Redis/文件） | 简单（共享内存），但要加锁防抢 |
| 隔离 | 强：一个崩不连累别的 | 弱：一个线程崩可能拖垮整进程 |

- **包含关系单向**：`进程 ⊃ 线程`。进程是大容器装线程；线程属于唯一进程，装不下进程。
- 类比：**进程 = 不同公司**（各自办公楼/资料柜，不互通）；**线程 = 同公司员工**（共用一栋楼）。

### Node 是单线程还是单进程？

- **准确说**：Node **用单线程执行 JS**（一个进程内只有一根 JS 主线程 = 事件循环线程）。这是"Node 单线程"的本意。
- **进程维度**：Node 程序就是一个进程，默认单进程，但 `cluster` 能 fork 多进程。说"单进程"不准确。
- **常见误解纠正**：「Node 单线程」≠「整个 Node 只有一根线程」。底层 **libuv 有线程池（默认 4 个）**，后台帮做文件读写/DNS/crypto。完整模型：**单线程执行 JS + 底层多线程做部分 I/O**。

```text
Node 进程
├── 1 个 JS 主线程（事件循环）  ← "单线程"指这个，你的代码都在这跑
└── libuv 线程池（默认4）        ← 后台做文件/DNS/crypto
```

### 三个词别混

- **单线程** → JS 执行（Node 默认特性）
- **cluster** → **多进程**（8 worker = 8 进程，内存不共享，图隔离性）
- **worker_threads** → **多线程**（多 JS 线程共享内存，适合 CPU 密集活）

### "单线程"为什么能 fork 多进程？（不矛盾）

两个维度不冲突：**"单线程"管的是进程内部；"fork"是在进程外部复制。**
fork 不是把一根线程劈成多根，而是**另起一个全新的独立 Node 进程**，这个新进程**自己也是单线程**。

```text
fork 前: [进程A: 1根线程]
fork 后: [进程A:1根][进程B:1根]...[进程H:1根]  ← 每个进程内部仍单线程，8进程=8根，互不共享内存
```

- 类比：不让 1 个员工变超人（多线程），而是**再开 8 家分公司**，每家 1 员工（多进程）。
- **谁来 fork？** 是**操作系统**的能力。master 喊 `cluster.fork()`，OS 复制出子进程——不是 JS 线程自己分裂。

### I/O vs CPU（速记）

**「算」的是 CPU，「等」的是 I/O。**

| CPU（自己算，占 CPU） | I/O（等外部，CPU 闲） |
|---|---|
| base64 编码、图片压缩/缩放、JSON 序列化、MD5/加密、gzip、大数组排序、大文本正则 | 等网站加载、等 Chrome 渲染、读写磁盘、查数据库、调 Redis |

→ 看到一个操作就问："它在埋头算，还是在等别人回话？"
